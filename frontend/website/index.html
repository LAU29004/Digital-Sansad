The Problem

Laws that govern you,
written so you can't read them.

Indian parliamentary bills routinely exceed 100,000 tokens and are written in dense legalese. Every citizen query naively fed into an LLM wastes enormous compute — and produces carbon. Two compounding crises: an accessibility gap and an environmental cost.

📜

Accessibility Gap

Citizens left uninformed about policies that directly affect their lives.

⚡

Environmental Cost

Feeding 100k+ token documents into LLMs on every query is unsustainable at scale.

Core Innovation

4-Layer Token
Compression Pipeline

The expensive work happens once at ingestion — not on every citizen query. Each bill is compressed through four NLP layers before storage, achieving 90–93% token reduction while preserving 95%+ semantic content.

L1

Structural Strip

Regex removal of gazette headers, enactment clauses, assent notices, secretary signatures, financial memorandums, bare page numbers.

~30% off
L2

Semantic Deduplication

all-MiniLM-L6-v2 encodes all sentences locally. Drops any sentence with cosine similarity ≥ 0.90 to an earlier one. Eliminates repeated boilerplate clauses common across Indian bills.

~55% off
L3

Extractive TF-IDF Ranking

Keeps top 20% of sentences by TF-IDF score. Legal signal words (shall, penalty, offence, liable, tribunal…) add +0.4 per hit. Minimum 8 sentences always retained.

~90% off
L4

Semantic Section Classification

Each sentence matched against 9 buckets: penalties · citizen_rights · regulatory_body · obligations · definitions · amendments · appeal_mechanism · schedules · enforcement.

Features

Built for every
Indian citizen.

A complete end-to-end platform — from automated bill monitoring to a plain-English AI chat interface on your phone.

🔄

Automated Sansad Monitoring

BlockingScheduler polls the official Sansad REST API every 6 hours for new Lok Sabha Government Bills. Runs immediately on startup.

📄

Dual PDF Extractor

pdfplumber first, PyMuPDF fallback. Both handle page-level failures gracefully without crashing the pipeline.

⛓️

Chain-of-Density

3 rounds of cheap-LLM rewriting pack more legal facts per sentence at query time — maximizing information density in the strong LLM's context window.

🔎

Semantic Search

ChromaDB retrieves the top-3 most relevant compressed sections per bill using cosine similarity on local MiniLM-L6-v2 embeddings. No API call needed.

🤖

Two-Model Architecture

Small 8B model (Llama-3.1) for high-frequency schema filling and CoD. Large 70B model (Llama-3.3) reserved for the single final citizen answer.

📤

PDF Upload

Citizens upload any government PDF. Compressed on-the-fly by the full 4-layer pipeline and answered by Gemini 2.0 Flash. Nothing stored.

📱

React Native App

Bill list with full-text search, expandable compressed section cards, AI chat interface, dark/light theme toggle via Expo.

📊

Token Compression Reports

Every ingestion logs raw → L1 → L2 → L3 token counts. Every query logs CoD before/after and overall compression ratio.

🔁

Smart Re-embedding

/trigger-ingest detects bills present in PostgreSQL but absent from ChromaDB and re-embeds from the locally cached PDF automatically.

Tech Stack

Production-grade
from day one.

Every technology chosen for a reason — performance, cost, or citizen accessibility.

Frontend

React Native + Expo — ExploreScreen, BillDetailsScreen, ChatScreen, PDF Upload. Runs on iOS, Android, and web.

Backend

FastAPI 0.111 — Async REST, CORS, auto OpenAPI/Swagger docs at /docs. Uvicorn ASGI server.

Vector DB

ChromaDB (persistent) — cosine similarity space. Sections keyed as {bill_number}::{section}. 384-dim MiniLM embeddings.

Relational DB

PostgreSQL — bills + bill_sections tables via SQLAlchemy 2.0 + pg_insert upsert. Alembic schema migrations.

Cheap LLM

Llama-3.1-8B-Instruct — schema extraction + Chain-of-Density (HuggingFace Inference API, Cerebras provider).

Strong LLM

Llama-3.3-70B-Instruct — final citizen answer ≤200 words (HuggingFace / Novita provider). Gemini 2.0 Flash for PDF uploads.

Scheduler

APScheduler BlockingScheduler — 6h poll interval, Asia/Kolkata timezone, max_instances=1 to prevent overlap.

Compression Engine

Custom 4-layer NLP: L1 regex, L2 MiniLM dedup, L3 TF-IDF (max_features=10,000), L4 keyword classify into 9 sections.
